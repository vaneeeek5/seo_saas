1. Назначение документа
Данный документ определяет единственную допустимую архитектуру платформы SEO OS.
Все последующие решения (код, API, UI, инфраструктура, базы данных, AI-агенты) должны соответствовать этому документу.
Если возникает противоречие между новым решением и архитектурой — изменяется предложение, а не архитектура.

2. Архитектурные принципы
Principle 1. Project First
В центре системы находится проект, а не статья.
Все сущности принадлежат проекту.
Organization
    │
    ▼
 Project
    │
    ├── Website
    ├── Semantic
    ├── Content
    ├── Publications
    ├── Analytics
    ├── Settings
    └── Tasks


Principle 2. Modular Monolith (MVP)
Это первое важное архитектурное решение.
Мы НЕ строим микросервисную архитектуру.
И НЕ строим один большой монолит.
Мы строим Modular Monolith.
Почему?
Потому что:
	•	проще разрабатывать;
	•	проще деплоить;
	•	проще тестировать;
	•	нет сетевых задержек между сервисами;
	•	можно легко перейти к микросервисам позже.
Каждый модуль имеет собственные:
	•	интерфейсы;
	•	бизнес-логику;
	•	модели;
	•	репозитории;
	•	API.
Но все они находятся в одном приложении.

Principle 3. Domain Driven Design
Каждый модуль отвечает только за свою область ответственности.
Например
Semantic Engine

ничего не знает

о Publisher

Publisher
ничего не знает

о генерации текста

Analytics
не знает

как устроена семантика

Это жесткое правило.

Principle 4. Event Driven
Внутри системы нет длинных цепочек вызовов.
Все построено на событиях.
Например
Project Created

↓

Website Connected

↓

Website Crawled

↓

Semantic Ready

↓

Clusters Ready

↓

Content Plan Ready

↓

Article Generated

↓

Images Generated

↓

Publication Finished

Любой модуль может подписываться на события.

Principle 5. Queue First
Все долгие операции выполняются через очередь задач.
Запрещено:
HTTP Request

↓

Ждать 5 минут

Правильно:
HTTP

↓

Task

↓

Queue

↓

Worker

↓

Result

4. Описание модулей
Core
Ядро платформы.
Отвечает за:
	•	конфигурацию;
	•	общие интерфейсы;
	•	DI;
	•	логирование;
	•	события;
	•	очередь;
	•	общие типы.
Core не содержит бизнес-логики.

Identity
Пользователи.
Организации.
Авторизация.
JWT.
Роли.

Projects
Самый главный модуль.
Именно здесь живет объект Project.
Все остальные сервисы работают через него.

Website Engine
Отвечает за:
	•	подключение сайта;
	•	проверку CMS;
	•	обход сайта;
	•	карту страниц;
	•	robots.txt;
	•	sitemap;
	•	определение структуры.

Semantic Engine
Самостоятельный сервис.
Функции:
	•	сбор семантики;
	•	очистка;
	•	нормализация;
	•	кластеризация;
	•	определение Intent;
	•	приоритизация.

Content Engine
Создает:

	•	заголовки;
	•	текст;
	•	FAQ;
	•	Schema;
	•	Meta.

Отдельный модуль.
Он никогда не пишет статью.

Media Engine
Создает:

	•	ALT;
	•	Caption;
	•	OG Image.

API

6. Adapter Pattern
Все внешние сервисы работают одинаково.
Например
Publisher Interface

↓

WordPress Adapter

↓

Tilda Adapter

↓

Custom API Adapter

То же самое относится к AI-моделям и источникам семантики.
Это позволит заменить поставщика или добавить нового без изменения остального кода.

7. Общие правила разработки
Каждый модуль обязан:
	•	иметь собственный публичный API;
	•	скрывать внутреннюю реализацию;
	•	не обращаться напрямую к внутренним классам других модулей;
	•	обмениваться данными через DTO и события;
	•	иметь независимые тесты.

Архитектурные решения (ADR)
Я предлагаю добавить в проект ещё один документ, которого сейчас нет, но который станет одним из самых важных.
Architecture Decision Records (ADR)
Вместо того чтобы просто принимать решения, мы будем фиксировать их.
Например:
ADR-001 — Используем Modular Monolith вместо микросервисов.
ADR-002 — Центром системы является Project.
ADR-003 — Все внешние интеграции реализуются через Adapter Pattern.
ADR-004 — Все долгие операции выполняются через очередь задач.
ADR-005 — Взаимодействие модулей основано на событиях.
Каждый ADR будет содержать:
	•	проблему;
	•	рассмотренные варианты;
	•	принятое решение;
	•	причины выбора;
	•	последствия;
	•	когда решение может быть пересмотрено.
Это очень ценно по двум причинам:
	•	Через полгода будет понятно, почему архитектура выглядит именно так.
	•	Antigravity, Codex и разработчики смогут опираться не только на итоговое решение, но и на его обоснование.
