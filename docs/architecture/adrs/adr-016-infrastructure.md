# ADR-016: Infrastructure & Deployment

## Статус
Принято

## Контекст
Платформа должна соответствовать принципу Cloud Native и быть Stateless (любой сервис может быть перезапущен без потери состояния). Нам необходимо определить базовые подходы к развертыванию приложения, управлению контейнерами и обработке загружаемых или генерируемых файлов (изображения, сгенерированные статьи).

## Решение
Устанавливаются следующие инфраструктурные стандарты:

1. **Контейнеризация:** Приложение (Frontend, Backend, Workers) должно быть полностью контейнеризировано с использованием Docker. Развертывание происходит исключительно через образы (Stateless подход).
2. **Работа с файлами (Медиа):** 
   - Сервер (Backend) никогда не сохраняет сгенерированные изображения или загружаемые файлы на локальный диск или в Volume-контейнеры.
   - Для хранения файлов используется облачное хранилище (AWS S3, Google Cloud Storage, Cloudflare R2).
   - Загрузка файлов клиентами или сохранение сгенерированных изображений происходит напрямую в Cloud Storage (используются Signed URLs для безопасной аутентификации загрузок с Frontend'а, минуя Backend для передачи тяжелых бинарников).
3. **Stateless архитектура:** Любая сессионная информация, кэш и данные о состоянии сохраняются в Redis (используется как для очередей BullMQ, так и для хранения кэша). Контейнеры Backend'а можно масштабировать горизонтально (множество инстансов) без конфликтов.

## Последствия
**Положительные:**
- Бесконечное и дешевое масштабирование хранилища медиа-файлов через S3/Cloud Storage.
- Отсутствие "узких горлышек" при развертывании и масштабировании Backend'а благодаря отсутствию привязки к локальному диску.
- Разделение ответственности: Backend занимается только логикой и координацией (генерация Signed URL), а не проксированием трафика для файлов.

**Отрицательные:**
- Повышенная сложность настройки сетевого доступа (CORS, Signed URL) к облачным бакетам.
- Зависимость от стороннего облачного провайдера для хранения файлов.
