# ADR-014: Frontend Realtime Communication

## Статус
Принято

## Контекст
Платформа выполняет множество длительных асинхронных процессов (генерация статей через LLM, парсинг сайтов, сбор семантики). Синхронные HTTP-запросы в таких условиях недопустимы (Principle 5. Queue First). Пользовательский интерфейс (UI) должен мгновенно реагировать на изменение статусов задач без необходимости ручного обновления страницы или ресурсоемкого long-polling'а со стороны клиента.

## Решение
В качестве стандарта общения между Backend и Frontend для асинхронных задач мы используем **Server-Sent Events (SSE)**.

1. **Инициализация задачи:** Frontend отправляет стандартный HTTP POST запрос (например, `POST /api/projects/1/generate-content`).
2. **Ответ Backend:** Backend немедленно возвращает `202 Accepted` и `taskId` (идентификатор задачи), после чего задача помещается в очередь (Task Engine).
3. **Подписка на обновления:** Frontend устанавливает SSE-соединение (например, `GET /api/tasks/{taskId}/events` или общую подписку на проект).
4. **Обновление UI:** Backend пушит события изменения статуса задачи (Pending, Processing, Completed, Failed) и частичные результаты (прогресс) через SSE. Frontend слушает эти события и реактивно обновляет интерфейс (progress bars, статусы).

## Последствия
**Положительные:**
- Однонаправленный поток данных (Backend -> Frontend) через SSE проще в реализации и масштабировании, чем WebSockets, так как нам в основном требуются уведомления от сервера.
- Снижение нагрузки на сеть по сравнению с polling'ом.
- Нативная поддержка в браузерах через API `EventSource`.

**Отрицательные:**
- Требуется аккуратная обработка обрывов соединений и автоматическое переподключение на клиенте.
- В случае масштабирования backend-инстансов потребуется pub/sub механизм (например, Redis Pub/Sub), чтобы транслировать события SSE на тот инстанс сервера, к которому подключен клиент.
