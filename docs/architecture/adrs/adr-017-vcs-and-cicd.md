# ADR-017: Version Control & CI/CD Pipeline

## Статус
Принято

## Контекст
С ростом функционала платформы критически важно стандартизировать процессы интеграции и доставки кода. Необходимо избежать ситуации с рассинхронизацией кода на серверах и локальных машинах разработчиков. Требуется автоматизировать проверки качества кода и процесс деплоя.

## Решение
1. **Single Source of Truth:** GitHub является единственным источником истины для кода. Прямые изменения кода на боевом или тестовом сервере (через SSH, FTP и прочее) строжайше запрещены.
2. **Процесс CI/CD (GitHub Actions):** 
   - Любой пуш в ветку `main` (или слияние Pull Request'а) должен автоматически запускать пайплайн GitHub Actions.
   - Пайплайн состоит из следующих обязательных шагов:
     1. Checkout кода.
     2. Установка зависимостей и запуск линтеров (ESLint, Prettier, TypeScript tsc).
     3. Сборка Docker-образов (Build).
     4. Деплой новой версии (Deploy) на целевой сервер (например, через SSH-перезапуск Docker Compose или обновление Cloud Run / ECS).

## Последствия
**Положительные:**
- Исключение человеческого фактора при деплое.
- Строгий контроль качества на этапе проверки PR, никто не может сломать ветку `main` (при включенных обязательных проверках).
- Быстрый откат (Rollback) к предыдущей версии, если билд окажется сломанным.

**Отрицательные:**
- Требуется время на поддержку и настройку CI/CD скриптов (`deploy.yml`) и секретов в GitHub.
- Деплой через CI/CD может быть немного дольше (из-за этапа сборки образов), чем просто "git pull" на сервере, но это компенсируется надежностью.
