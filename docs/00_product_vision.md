SEO Content Factory
Product Vision

1. Миссия проекта
SEO Content Factory — это AI-платформа полного цикла для автоматизированного создания, публикации и развития SEO-контента.
Главная цель платформы — сократить путь от идеи статьи до её публикации и дальнейшего роста органического трафика с нескольких часов или дней до нескольких минут, при этом сохраняя качество, масштабируемость и контроль над процессом.
Платформа должна выполнять роль не просто генератора текстов, а полноценного SEO-специалиста, который самостоятельно проходит весь цикл работы с контентом:
	•	анализирует сайт;
	•	определяет перспективные темы;
	•	собирает и кластеризует семантику;
	•	составляет контент-план;
	•	пишет статьи;
	•	генерирует изображения;
	•	публикует материалы;
	•	анализирует результаты;
	•	предлагает дальнейшие действия.

2. Проблема рынка
Сегодня процесс SEO-продвижения состоит из большого количества разрозненных инструментов.
Типичный процесс выглядит следующим образом:
Ahrefs

↓

Wordstat

↓

Google

↓

Google Docs

↓

ChatGPT

↓

Midjourney

↓

Canva

↓

WordPress

↓

Search Console

↓

GA4

Каждый этап выполняется отдельно.
Человек вручную переносит информацию между сервисами.
Из-за этого:
	•	процесс дорогой;
	•	занимает много времени;
	•	плохо масштабируется;
	•	зависит от человеческого фактора;
	•	сложно автоматизируется.

3. Решение
SEO Content Factory объединяет весь цикл в единую платформу.
Пользователь работает только в одном интерфейсе.
Все остальные действия выполняют специализированные AI-модули.

4. Философия продукта
Есть несколько принципов, которые нельзя нарушать.
4.1 AI First
Все действия сначала проектируются как автоматические.
Ручной режим добавляется только при необходимости.

4.2 Modular
Каждая подсистема является независимым сервисом.
Никакой модуль не должен знать внутреннюю реализацию другого.
Взаимодействие происходит исключительно через публичные интерфейсы.

4.3 API First
Любая возможность платформы должна существовать как API.
Интерфейс кабинета — всего лишь один из клиентов этого API.

4.4 Cloud Native
Платформа изначально проектируется для работы в облаке.
Локальный запуск допускается только для разработки.

4.5 Stateless
Любой сервис может быть перезапущен без потери состояния.
Все данные хранятся вне приложений.

4.6 Event Driven
Платформа реагирует на события.
Например:
Проект создан

↓

Начать анализ

↓

Семантика готова

↓

Создать статьи

↓

Статья готова

↓

Создать изображения

↓

Изображения готовы

↓

Опубликовать

Никаких длинных синхронных процессов.

5. Главная цель платформы
Не стать ещё одним AI Writer.
И не стать ещё одной CMS.
Главная цель:
Создать систему, которая самостоятельно занимается развитием SEO-проекта.

6. Что НЕ является целью
Платформа НЕ должна становиться:
	•	визуальным конструктором сайтов;
	•	CMS;
	•	редактором изображений;
	•	аналогом Photoshop;
	•	аналогом Figma;
	•	сервисом для написания рекламных текстов;
	•	SMM-платформой.
Все это можно интегрировать позже.

7. Целевая аудитория
Первый сегмент
SEO-специалисты.

Второй сегмент
Маркетинговые агентства.

Третий сегмент
Владельцы сайтов.

Четвертый сегмент
Интернет-магазины.

Пятый сегмент
Контент-команды.

Шестой сегмент
SaaS-компании.

8. Конкурентное преимущество
Большинство существующих сервисов автоматизируют только отдельный этап процесса.
Например:
	•	только генерацию текста;
	•	только анализ семантики;
	•	только анализ позиций;
	•	только публикацию.
SEO Content Factory объединяет эти этапы в единую систему.
Главным объектом управления становится не статья, а SEO-проект целиком.

9. Основной объект платформы
Не статья.
Не ключ.
Не изображение.
Основной объект:
Project

Все остальное является частью проекта.
Project

├── Website

├── Semantic

├── Articles

├── Images

├── Publications

├── Analytics

└── Settings


10. Масштабируемость
Архитектура платформы должна позволять без изменения ядра:
	•	подключать новые CMS;
	•	подключать новые AI-модели;
	•	подключать новые поисковые системы;
	•	подключать новые сервисы аналитики;
	•	подключать новые генераторы изображений;
	•	подключать новые языки;
	•	подключать новых AI-агентов.

11. Этапы развития
Phase 1
Рабочий MVP.
Поддержка:
	•	WordPress;
	•	Tilda;
	•	Custom API.

Phase 2
Расширение возможностей.
Добавляются:
	•	аналитика;
	•	дополнительные CMS;
	•	новые источники семантики;
	•	новые AI-модели.

Phase 3
Автономная SEO-платформа.
Появляются:
	•	AI-агенты;
	•	автоматическая переоптимизация;
	•	рекомендации;
	•	прогнозирование;
	•	автоматическое развитие проекта.

12. Definition of Success
Платформа считается успешной, если пользователь может:
	•	Подключить сайт.
	•	Получить семантику.
	•	Получить готовый контент-план.
	•	Сгенерировать статьи.
	•	Получить изображения.
	•	Опубликовать материалы.
	•	Отслеживать результаты.
	•	Развивать сайт без использования сторонних SEO-сервисов для большинства типовых задач.

Product Requirements Document (PRD)
Версия 1.0

1. Product Overview
Назначение
SEO OS — облачная SaaS-платформа, которая автоматизирует полный жизненный цикл SEO-проекта.
Пользователь работает только с проектом.
Все остальные действия выполняются автоматически или по запросу.

2. Основной объект системы
Это, наверное, самое важное решение во всей архитектуре.
Большинство сервисов строятся вокруг статьи.
Мы строим вокруг проекта.
Project

↓

Website

↓

Semantic

↓

Content

↓

Publishing

↓

Analytics

↓

Growth

То есть пользователь никогда не работает отдельно со статьёй.
Он работает с проектом.

3. Жизненный цикл проекта
После создания проекта платформа проходит следующие стадии.
Проект

↓

Подключение сайта

↓

Анализ сайта

↓

Сбор семантики

↓

Кластеризация

↓

Построение карты тем

↓

Контент-план

↓

Создание статей

↓

Генерация изображений

↓

Публикация

↓

Мониторинг

↓

Рост проекта

Это становится главным Pipeline платформы.

4. Основные пользовательские сценарии
Сценарий №1
Создание нового проекта
Пользователь:
	•	вводит URL;
	•	выбирает CMS;
	•	подключает сайт.
После этого запускается автоматический анализ.

Сценарий №2
Получение семантики
Платформа:
	•	анализирует сайт;
	•	определяет тематику;
	•	получает ключевые запросы;
	•	очищает мусор;
	•	определяет intent;
	•	создаёт кластеры.

Сценарий №3
Создание контент-плана
Из кластеров автоматически строится структура будущего сайта.
Например:
Категория

↓

Подкатегория

↓

Статья

↓

FAQ

↓

Связанные материалы

Не просто список статей, а карта знаний.

Сценарий №4
Создание статьи
Для каждой темы система:
	•	определяет структуру;
	•	пишет статью;
	•	проводит Humanizer;
	•	генерирует Title;
	•	генерирует Description;
	•	создаёт FAQ;
	•	создаёт Schema;
	•	создаёт изображения.

Сценарий №5
Публикация
Статья проходит проверку.
После подтверждения пользователь может:
	•	опубликовать сразу;
	•	поставить в очередь;
	•	запланировать публикацию.

Сценарий №6
Развитие проекта
После публикации система продолжает работать.
Она должна видеть:
	•	индексируется статья или нет;
	•	появились ли позиции;
	•	есть ли показы;
	•	нужен ли апдейт;
	•	нужно ли написать ещё материалы.

5. Минимальный MVP
Очень важно ограничить первую версию.
В MVP входят только:
CMS
	•	WordPress
	•	Tilda
	•	Custom API

Семантика
	•	Яндекс Вордстат
	•	Google Suggestions

Генерация
	•	Gemini
	•	Nano Banana

AI
	•	Humanizer

Аналитика
Только базовая:
	•	статус проекта;
	•	статус статьи;
	•	статус публикации.
Никаких Search Console в MVP.

6. Пользовательские роли
На мой взгляд, MVP должен поддерживать только две роли:
Owner
Полный доступ.

Editor
Работа с контентом.
Без изменения проекта.
Никаких сложных RBAC пока.

7. Что не входит в MVP
Важно зафиксировать сразу.
Не входит:
	•	биллинг;
	•	агентская структура;
	•	White Label;
	•	командная работа;
	•	Search Console;
	•	GA4;
	•	Яндекс Метрика;
	•	Bitrix;
	•	Ghost;
	•	Drupal;
	•	массовый импорт сайтов;
	•	AI-чат.

8. KPI MVP
Через платформу пользователь должен иметь возможность:
	•	подключить сайт менее чем за 3 минуты;
	•	получить семантику менее чем за 5 минут;
	•	получить первую статью менее чем за 3 минуты;
	•	получить публикацию менее чем за 10 секунд после готовности материала.

Архитектурное решение №1 (предлагаю зафиксировать)
Я бы полностью отказался от понятия "статья" как центрального объекта интерфейса.
Практически все SEO-сервисы строят интерфейс вокруг статей.
Мне кажется, для нашей платформы сильнее строить всё вокруг карты проекта.
То есть пользователь видит не список из 500 статей, а дерево:
Проект
│
├── Семантика
│   ├── Кластеры
│   ├── Темы
│   └── Потенциал
│
├── Контент
│   ├── Запланировано
│   ├── В работе
│   ├── Готово
│   └── Опубликовано
│
├── Публикации
│
├── Аналитика
│
└── Развитие

Из этого решения потом вырастет и фронтенд, и API, и архитектура модулей.
